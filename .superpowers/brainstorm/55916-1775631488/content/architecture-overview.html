A2A Architecture — Query Flow

New architecture: Triage → OrchestratorAgent → A2A preprocessing → execution

Current Flow (monolith)

How it works today
User Query → routing_agent.py (1200 LOC)
↳ GLiNER entity extraction (inline)
↳ SpaCy relationship analysis (inline)
↳ SIMBA query enhancement (inline)
↳ QueryEnhancementPipeline (inline)
↳ DSPy routing decision (inline)
↳ GRPO optimization (inline)
→ if simple → search_agent
→ if complex → MultiAgentOrchestrator (in-process) → agents

New Flow (A2A)

Proposed architecture
User Query → Triage (GLiNER fast path, <100ms)
SIMPLE → search_agent / summarizer / etc.
COMPLEX → OrchestratorAgent (A2A)
LLM planner decides steps dynamically:
1 EntityExtractionAgent (GLiNER + SpaCy + LLM)
2 QueryEnhancementAgent (SIMBA + DSPy)
3 RoutingAgent (thin DSPy decision only)
4 ProfileSelectionAgent (Vespa profile picker)
5 Execution agents (search, summarizer, report, etc.)
+ streaming, checkpoints, cross-modal fusion, workflow intelligence, cancellation

What changes

Deleted / gutted

  • routing_agent.py — gutted from 1200→~200 LOC
  • MultiAgentOrchestrator — deleted entirely
  • Inline GLiNER/SpaCy/SIMBA in routing_agent — deleted
  • query_enhancement_engine.py — moves into agent

Enhanced / new

  • OrchestratorAgent — gains streaming, checkpoints, fusion, intelligence, cancellation
  • EntityExtractionAgent — gains GLiNER + SpaCy
  • QueryEnhancementAgent — gains SIMBA + pipeline
  • Triage — extracted from ComprehensiveRouter fast path
  • config_loader.py + agent_dispatcher.py — rewired
