A2A Architecture Restructuring — Full Design

Finalized design: telemetry-first, Argo batch optimization, clean A2A agent separation

1. Architecture Overview

Query Flow — New Architecture
User Query → GatewayAgent (GLiNER, <100ms)
SIMPLE → execution agent directly (search, summarizer, etc.)
COMPLEX → OrchestratorAgent
LLM planner decides steps dynamically:
1 EntityExtractionAgent (GLiNER + SpaCy + LLM)
2 QueryEnhancementAgent (DSPy + optimized module from artifacts)
3 RoutingAgent (thin DSPy decision + learned optimizer)
4 ProfileSelectionAgent (Vespa profile picker)
5 Execution agents (search, summarizer, report, etc.)
+ streaming, checkpoints, cross-modal fusion, workflow intelligence, cancellation

3. Telemetry-First Learning Architecture

Agents emit spans. Argo batch jobs read spans and optimize. No inline optimization loops.

Request Path vs Batch Path

Request Path (agents)

1. Agent does work
2. Emits telemetry span with attributes
3. Returns result
What agents load at startup:
• Optimized DSPy modules (from artifacts)
• Workflow templates (from artifacts)
• Strategies from Mem0 (via MemoryAwareMixin)

What agents do NOT do:
• No SIMBA.compile()
• No DSPy teleprompter optimization
• No in-memory pattern banks
• No inline recording to separate stores

Batch Path (Argo CronWorkflows)

1. Read spans from Phoenix
2. QualityMonitor scores them
3. Run optimizations
4. Save artifacts + strategies
Optimization jobs:
• SIMBA: Read enhancement spans → build training set → compile DSPy module → save artifact
• Routing: Read routing spans → GEPA/MIPROv2 → save optimized module
• Workflow: Read orchestration spans → generate templates → save artifacts
• Strategy: Read scored spans → distill strategies → store in Mem0
• Thresholds: Read routing spans → update adaptive thresholds → save config

EntityExtractionAgent cogniverse.entity_extraction query, entities, relationships, entity_count Entity extraction optimizer

4. Memory & Learning Systems

Mem0 Persistent Memory

What: Per-tenant, per-agent interaction history backed by Vespa
Used by: Any agent via MemoryAwareMixin
Request path: get_relevant_context(query) + inject_context_into_prompt()
Contains: Past interactions, successes, failures
Architecture change: None — works via mixin, moves with each agent

Strategy Learner

What: Distills Phoenix traces into reusable strategies
Input: Phoenix scored spans (NOT Mem0)
Output: Strategies stored in Mem0
Runs: Argo batch job (optimization_cli --mode triggered)
Retrieved by: Any agent via get_strategies(query)
Architecture change: None — already a batch job

DSPy Optimizers (GEPA/MIPROv2/SIMBA/Bootstrap)

What: Compile optimized DSPy modules from training data
Input: Phoenix spans (query enhancement, routing, orchestration)
Output: Optimized DSPy module artifacts
Runs: Argo batch job
Loaded by: Agents at startup from artifact store
Change: SIMBA inline optimization REMOVED from QueryEnhancementAgent. All compilation is batch-only.

Workflow Intelligence

What: Learns orchestration patterns from past workflows
Input: Phoenix orchestration spans
Output: Workflow templates + agent performance profiles (artifacts)
Runs: Argo batch job
Loaded by: OrchestratorAgent at startup
Change: Inline DSPy optimization REMOVED. OrchestratorAgent only loads templates, never runs optimization.

7. Key Design Decisions

Decision Choice Rationale
Entry point model Tiered: GatewayAgent triage GLiNER fast path for simple queries, full orchestration for complex
Orchestrator planning Full LLM-planned pipeline Dynamic agent selection per query, not fixed sequence
RoutingAgent separate? Yes, keep separate 2,300 LOC of routing-specific optimization (GEPA/MIPROv2) would be lost if folded
Learning architecture Telemetry-first Agents emit spans, Argo batch jobs optimize. No inline optimization loops.
SIMBA inline pattern bank Eliminated Anti-pattern: SIMBA.compile() in request path. Batch-only via Argo.
WorkflowIntelligence inline optimization Eliminated Anti-pattern: DSPy optimization in request path. Batch-only via Argo.
Migration strategy Big bang Not production — build new, swap everything at once
MultiAgentOrchestrator features All 5 ported to OrchestratorAgent Streaming, checkpoints, fusion, workflow intelligence, cancellation
